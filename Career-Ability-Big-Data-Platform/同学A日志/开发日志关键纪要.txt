第3天：ETL 清洗脚本（最复杂任务）+ Bug修复
================================================================================

【完成内容】
1. 编写 data-pipeline/etl_clean.py（~490行，常驻进程）
   对照前期规划文档 7.3 节逐项实现了8个清洗步骤：

   ① BRPOP 阻塞消费 queue:raw-job-data（超时5秒，支持优雅退出）
   ② 空值校验：title 或 company.name 为空则丢弃
   ③ 薪资标准化：年薪转月薪（>80除以12）、min/max 顺序修正
      （注：原始文本解析如 "8K-15K" 由 import_data.py 预处理，ETL做二次校正）
   ④ 城市标准化：查 city_mapping.json，三级模糊匹配（直接→去"市"→加"市"）
   ⑤ 学历标准化：中英文关键词 → 不限/大专/本科/硕士/博士
   ⑥ 经验标准化：中英文关键词 → 不限/应届/1-3年/3-5年/5-10年/10年以上
   ⑦ 技能提取：skill_dict.json 关键词在 title+description 中全文匹配，
      处理别名映射，保留 iOS/gRPC/CI/CD 等专有名词大小写
   ⑧ MD5去重：jobId + sourceUrl 计算MD5，查MySQL跳过重复记录
      写MySQL：先查/插 job_company，拿 company_id，再 INSERT job_position

   附加功能：产入清洗队列 queue:cleaned-job-data（供C同学消费）、
   统计计数器、批量进度日志、SIGINT/SIGTERM 优雅退出、异常事务回滚

2. 代码审查与 Bug 修复（6项）

   【严重Bug — 已修复】
   Bug #1: normalize_experience 中 "5-10年" 被误判为 "10年以上"
   - 原因：检查组 "10年以上" 的关键词 "10" 是 "5-10年" 的子串
   - 修复：将 "5-10年" 范围检查提到 "10年以上" 之前，避免子串干扰

   Bug #2: normalize_experience 中单数字匹配过于宽泛
   - 原因："6"/"7"/"8"/"9"/"4"/"2" 用 Python "in" 匹配会误命中 "16年"/"14年"
   - 示例："16年" 应归类为 "10年以上"，但被 "6" in "16年" 匹配为 "5-10年"
   - 修复：重构为分层策略——范围模式→中文数字词→英文→re.findall提取数字做区间判断

   【中等Bug — 已修复】
   Bug #3: import_data.py parse_salary 面议检查在空格清除之前
   - 原因：先检查面议(含空格敏感)，后去除空格
   - 示例："面 议"（带空格）不会被识别
   - 修复：将面议检查移到空格/单位清理之后，并加 lower()

   Bug #4: config.py 默认CSV路径错误
   - 原因：DEFAULT_CSV_PATH 指向 "../data/sample_jobs.csv"，实际文件是 test_sample.csv
   - 修复：改为 "../data/test_sample.csv"

   【轻度Bug — 已修复】
   Bug #5: normalize_salary 对非整数输入不安全
   - 原因：int() 失败后保留原始字符串，后续字符串与整数比较会抛 TypeError
   - 修复：int() 失败时将值置为 None

   Bug #6: BATCH_SIZE 语义混用
   - 原因：config.py 定义的 BATCH_SIZE（导入进度间隔）被 etl_clean.py 复用为日志间隔
   - 修复：etl_clean.py 单独定义 LOG_INTERVAL = 100

3. 全链路测试验证
   - 环境：Docker (MySQL 8.0 + Redis 7) + Python 3.13
   - 流程：CSV(20条) → import_data.py → Redis List → etl_clean.py → MySQL
   - 结果：20/20 成功写入，0 丢弃，0 异常
   - 去重验证：20/20 重复数据被正确跳过
   - 城市标准化：杭州→浙江/新一线，北京→北京/一线 等全部正确
   - 薪资标准化：15K-25K 等全部正确
   - 经验标准化：5-10年正确保留，不再误判为10年以上
   - 公司去重：20条记录中字节跳动出现2次，正确合并为1条 → 19家独立公司
   - 技能提取：每条记录4-6个技能标签

【关键要点】
- ETL 是前期核心里程碑——它打通了 CSV → Redis → MySQL 的完整数据链路，
  解除了同学C的阻塞（他需要 MySQL 中有真实数据才能测试岗位查询接口）
- normalize_experience 的 Bug 修复采用了分层策略：
  ① 范围的明确关键词（如"5-10年"中的"5-10"）优先级最高
  ② 中文数字词（十年/五年/三年/四年/一年/两年）
  ③ 英文关键词
  ④ 用 re.findall(r'\d+', s) 提取所有数字，取最大值做区间判断
  这比直接 in 匹配更精确，且新增年数经验格式也能自动适配
- 所有标准化函数都设计为"在 ETL 阶段可独立运行"，不依赖 import 阶段的
  预处理——这对数据可追溯性很重要
- 第3天完成后，C（张博源）的岗位查询接口和D（王致远）的报告推荐系统
  都有了真实数据支撑
