> 生效日期：2026-07-14
>
> 目标分支：`release/1.0.0-integration`
>
> 任务源：`完整项目合并与发行任务清单.md`

## 1. 台账结构

- GitHub [Milestone `v1.0.0`](https://github.com/wzt521521/Career-Ability-Big-Data-Platform/milestone/1) 统一收录本次发行任务。
- 80 个任务各对应一个 GitHub Issue，Issue 标题必须以 `[任务ID]` 开头。
- 已关闭任务保留 Issue 作为审计记录；未完成任务保持打开，直到完成定义全部满足。
- Issue 使用 `priority:P0`、`priority:P1`、`priority:P2` 和 `workstream:<分组>` 标签。
- Markdown 任务清单决定范围和优先级，GitHub Issue 记录负责人、讨论、依赖、PR 和验收证据；两者使用同一个任务 ID 关联。

## 3. Issue 规则

1. 一个 Issue 只对应一个任务 ID；不得把多个 ID 合并为无法独立验收的大任务。
2. Issue 必须包含范围、验收标准、验证命令、依赖和风险；P0 必须说明失败与回滚路径。
3. 状态变化、阻断原因和验收证据记录在 Issue，不以聊天记录替代。
4. 只有代码/文档经 PR 合入发行分支、自动化测试或可重复脚本通过、配置和回滚说明同步后才能关闭。
5. P2 移入 `v1.1` 必须保留 Issue 和决策记录，不得直接删除任务。

## 4. 分支与 PR 规则

- 从最新 `release/1.0.0-integration` 创建短生命周期分支。
- 分支格式：`fix/<task-id>-<topic>`、`feat/<task-id>-<topic>`、`docs/<task-id>-<topic>` 或 `chore/<task-id>-<topic>`。
- PR 标题格式：`[TASK-ID] type: summary`，正文使用仓库 PR 模板并通过 `Closes #<issue>` 关联 Issue。
- 一个 PR 只解决一个任务 ID；共享的机械修改应先拆分依赖 Issue，不得再次直接推入主线。
- PR 必须通过 `backend`、`frontend`、`data-pipeline`、`compose`，并解决所有审查会话；仓库所有者单人最终决策模式不要求独立批准。
- PR 可由 `@SunLe-01` 或 `@wzt521521` 在本机创建；`@wzt521521` 在必需检查通过后负责终审和合并，不再请求其他协作者参与审批。
- 合并顺序由依赖关系和 P0/P1 优先级决定；禁止因“改动很小”绕过门禁。

## 5. 完成定义

- 变更已通过 PR 合入发行分支。
- 自动化测试或可重复验收脚本覆盖成功、失败和权限边界。
- 在干净环境可复现，不依赖开发者机器残留状态。
- 文档、环境变量、数据库迁移、部署和回滚配置同步更新。
- Issue 中记录验证命令、关键结果、合入 PR 和残余风险。

## 6. 查询方式

- 全部发行任务：[Milestone `v1.0.0`](https://github.com/wzt521521/Career-Ability-Big-Data-Platform/milestone/1)
- 阻断 RC：Label `priority:P0` 且状态 Open
- 阻断正式版：Label `priority:P1` 且状态 Open
- 按模块：Label `workstream:R1`、`workstream:A` 至 `workstream:REL`
- 精确任务：在 [GitHub Issues](https://github.com/wzt521521/Career-Ability-Big-Data-Platform/issues) 搜索标题中的 `[R1-01]` 等任务 ID
