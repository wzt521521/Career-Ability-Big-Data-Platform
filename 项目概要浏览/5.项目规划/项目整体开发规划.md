# 职业能力大数据服务平台 — 项目开发规划

> 规划依据：[项目介绍](../1.项目介绍/项目理解.md)、[需求大纲概要](../2.需求说明书/需求大纲概要.md)、[架构设计文档](../3.架构设计/架构设计文档.md)
>
> 核心原则：**基础方案 5 个容器跑通全链路（数据导入 → 清洗 → 分析 → 展示 → 推荐），大数据组件全部选做。**
>
> 分工原则：**按业务领域纵向切分，每人负责相关功能的完整链路（前端 + 后端），减少跨领域沟通。**

## 二、分期计划

> 分期按**逻辑依赖关系**划分，不设固定时间节点。每个阶段有明确的完成标志，前一阶段未完成则不进入下一阶段。

### 前期 — 基础搭建 + 核心数据链路跑通

**目标：5 个容器能启动，数据能从 CSV 一路流入 MySQL，登录认证可用。前期只做"让数据跑通"的最小集合，不为业务功能铺张。**

| 编号 | 任务 | 所属模块 | 关键交付物 | 负责 |
|------|------|----------|-----------|------|
| P0-1 | MySQL 全部基础表建表 | 1 | `sql/init.sql`，含所有表的 DDL | A |
| P0-2 | Redis 配置（缓存 + List 队列 + 持久化） | 4/15 | Redis 启动即用，两个 List Key 就绪 | A |
| P0-3 | Python CSV 导入脚本 + 标准 JSON 输出 | 3 | `data-pipeline/import_data.py`，LPUSH 到 `queue:raw-job-data` | A |
| P0-4 | Python ETL 清洗脚本（标准化 + 技能提取 + 去重 + 写 MySQL） | 5 | `data-pipeline/etl_clean.py`，BRPOP → 清洗 → LPUSH + INSERT | A |
| P0-5 | Docker Compose 编排 5 容器 + Nginx 反向代理 | 10 | `docker-compose.yml`，`docker compose up` 一键启动全栈 | A |
| P0-6 | Spring Boot 项目骨架 + 多模块划分 + JPA 基础配置 | — | `backend/` 工程可启动，各业务模块包结构就绪 | B/C/D 各自 |
| P0-7 | 用户认证 API（登录/注册/Token 刷新/获取当前用户） | 2 | Postman 可调通认证全流程（JWT 签发 + 校验 + 刷新） | B |
| P0-8 | Vue 项目骨架 + 路由 + Axios 封装 + 登录页 + 401 拦截 | 2 | 前端可登录并跳转到空白主页，Token 自动附加 | B |
| P0-9 | 基础岗位查询 API（分页列表 / 详情 / 简单搜索） | 6/8 | 岗位 CRUD 接口可用，能查 MySQL 中的清洗后数据 | C |
| P0-10 | 学生画像 CRUD API（创建/编辑/查询画像） | 9 | 画像接口可调通，为推荐引擎准备输入数据 | D |
| P0-11 | 寻找并导入一份优质样本数据集 | — | 至少 500+ 条岗位数据入库，覆盖多个城市和岗位类型 | A |

> **前期完成标志：** 导入一份样本数据集（如 Kaggle 某城市招聘 CSV），`docker compose up` 一键启动全栈，登录后通过 Postman 或浏览器能查到 MySQL 中的岗位数据。**核心数据链路闭环：CSV → Redis List → ETL 清洗 → MySQL → API 查询。**
>
> **前期不包含：** 数据大屏、岗位分析页面、推荐引擎、报告生成、系统管理。这些属于中期。

## 三、四人分工（按业务领域纵向切分）

> **核心思路：** 每人负责一个完整业务领域的全栈开发（前端 + 后端），深入理解该领域的业务逻辑，避免一人对接多个业务方向导致的上下文切换和沟通成本。每个人只需要和"上游提供数据的人"以及"下游消费数据的人"沟通一次接口即可。

### 3.1 分工总览

| 角色 | 业务领域 | 一句话定位 | 技术栈 |
|------|----------|-----------|--------|
| **同学 A** | 数据管道与基础设施 | 让数据进得来、存得住、跑得通 | Python 3 + MySQL + Redis + Docker + Nginx + Spring Boot（采集管理部分） |
| **同学 B** | 用户权限与 API 体系 | 谁可以访问什么、外部如何接入 | Spring Boot + Spring Security + JWT + Redis + Vue 3（认证/管理相关页面） |
| **同学 C** | 数据分析与可视化 | 数据里有什么规律、如何展示 | Spring Boot + JPA + Redis + Vue 3 + ECharts |
| **同学 D** | 报告生成与岗位推荐 | 数据如何变成决策、如何服务个人 | Spring Boot + Freemarker + iText + Vue 3 + ECharts |

### 3.2 各角色详细职责

#### 同学 A：数据管道与基础设施

负责从原始数据到 MySQL 的全链路，以及开发/运行环境。

| 层级 | 模块编号 | 具体内容 |
|------|----------|----------|
| 数据处理 | 3.1 数据采集 | Python CSV/Excel 导入脚本 → 标准 JSON → LPUSH 到 Redis List |
| 数据处理 | 3.2 消息队列 | Redis List `queue:raw-job-data` / `queue:cleaned-job-data` 管理与对接 |
| 数据处理 | 3.3 ETL 清洗 | Python 清洗脚本：空值处理 / 薪资标准化 / 城市标准化 / 学历标准化 / 经验标准化 / 技能提取（500+ 词典）/ MD5 去重 → 写 MySQL |
| 后端 | 2.5 采集管理 | 数据源配置 / 任务 CRUD / `@Scheduled` Cron 调度 / 监控统计 / 失败重试（指数退避） |

### 3.4 协作节点与依赖

| 顺序 | 协作内容 | 依赖关系 | 说明 |
|------|----------|----------|------|
| 1 | A 交付建表脚本 + 样本数据 | B、C、D → A | 所有人等 A 的 MySQL 表结构才能写 JPA 实体。A 应最优先完成 |
| 2 | A 交付 Docker 环境 | B、C、D → A | 所有人统一用 Docker 开发，避免"我电脑能跑"问题 |
| 3 | B 交付认证 API + JWT 工具类 | C、D → B | C 和 D 的后端接口需要集成 Security 鉴权；各前端页面需要 Token 机制 |
| 4 | C 交付数据分析 API | D → C | D 的报告生成需要调用 C 的分析接口获取统计数据 |
| 5 | C、D 各自的前后端联调 | 内部闭环 | C 和 D 各负责自己领域的前后端，接口自定自调，无需跨人员协调 |
| 6 | B 的权限体系接入各模块 | C、D → B | B 提供注解和配置，C、D 在自己模块的方法上添加权限注解即可 |

> **关键依赖说明：** 前期阻塞点主要在 A（建表 + Docker）和 B（认证机制）。A 和 B 前期应集中精力完成各自的基础交付物，C 和 D 在 A 建表完成后即可并行推进各自的 JPA 实体和基础 API，在 B 交付认证后再统一接入 Security。

## 四、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 公开数据集质量差/字段缺失 | ETL 清洗效果不好 | 前期花半天挑选一份字段齐全的数据集（建议用 Kaggle 上评分较高的招聘数据集），必要时手动补全关键字段 |
| A 的建表和 Docker 进度滞后 | B、C、D 全部阻塞 | A 最优先完成建表脚本和 Docker Compose，其他三人可先写代码骨架，待 A 交付后再对接联调 |
| 报告生成 (PDF) 复杂度超预期 | 中期进度阻塞 | 前期优先跑通核心链路，报告模块预留充足时间；如时间紧，先用 HTML 转 PDF 的简单方案替代 Freemarker + iText |
| 推荐算法效果不理想 | 推荐结果看起来随机 | 先确保匹配逻辑无 bug，再调整权重参数；必要时降低对技能词典质量的依赖 |
| 四人技术背景不均衡 | 个别模块进度滞后 | 分工已按"全栈闭环"设计，每人负责完整的小领域，可根据个人强项微调（如 Python 熟的做 A，Java 熟的做 B/C/D）；后期 A 可作为机动力量支援 |
| Docker 环境问题 | 环境不一致导致联调困难 | A 在前期第一时间确定 Compose 文件，全员统一用 Docker 开发，不在宿主机直接运行服务 |
| C 和 D 的前端风格不一致 | UI 体验割裂 | 前期统一确定 Element Plus 组件使用规范和页面布局模板（B 的登录页作为第一个前端页面，可沉淀公共布局组件供 C、D 复用） |
