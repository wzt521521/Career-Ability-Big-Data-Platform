# 前期开发规划 — 基础搭建 + 核心数据链路跑通

> 前期定位：**只做"让数据跑通"的最小集合**。不涉及数据大屏、岗位分析页面、推荐引擎、报告生成、系统管理等中期内容。本阶段是项目整体开发的第一阶段，目标为后续中期业务功能开发提供可运行的基础设施、数据链路和认证体系。

---

## 一、阶段目标

### 1.1 一句话目标

**5 个容器能启动，数据从 CSV 一路流入 MySQL，登录认证可用。**

### 1.2 完成标志（验收标准）

1. `docker compose up` 一键启动 MySQL + Redis + Spring Boot + Vue (Nginx) + Python ETL 五个容器
2. 执行 Python 导入脚本，将一份 >= 500 条的样本数据集导入 Redis List
3. ETL 脚本自动消费、清洗、去重后写入 MySQL `job_position` / `job_company` 表
4. 通过 Postman 调用登录接口获取 JWT Token，再用 Token 调用岗位查询接口返回分页数据
5. 浏览器访问 `http://localhost` 可打开登录页，登录后跳转到空白主页（有导航骨架即可，不做业务页面）
6. 画像 CRUD 接口可正常创建/编辑/查询学生的就业画像

### 1.3 前期不包含（明确排除）

- 数据大屏（1.2）
- 岗位分析前端页面（1.3）
- 报告生成与报告中心（1.4）
- 岗位推荐引擎与推荐页面（1.5，仅做画像 CRUD 为后期准备数据）
- 系统管理前端页面（1.6，仅做后端认证）
- API 网关与文档（1.7/2.6）
- RBAC 权限细粒度控制（前期仅区分登录/未登录）
- Redis 缓存体系（除 List 队列外）
- 离线分析定时任务（3.4）

## 二、整体数据流

前期要跑通的核心链路如下，**每个箭头都必须验证通过**：

```
Kaggle CSV 样本数据集
        │
        ▼
[3.1 Python 导入脚本] ──→ 读取 CSV → 列映射 → 标准 JSON
        │
        ▼
[3.2 Redis List] queue:raw-job-data ← LPUSH 每条 JSON
        │
        ▼
[3.3 Python ETL 清洗] ← BRPOP 逐条消费
        │
        ├─ 空值校验（岗位名/公司名为空 → 丢弃）
        ├─ 薪资标准化（统一月薪 K）
        ├─ 城市标准化（省份 + 城市层级）
        ├─ 学历标准化（统一枚举）
        ├─ 经验标准化（统一枚举）
        ├─ 技能提取（500+ 技术词典匹配）
        └─ 去重（jobId + sourceUrl → MD5）
        │
        ▼
[4.1 MySQL] job_position / job_company ── 清洗后数据入库
        │
        ▼
[2.2 岗位查询 API] ← C 同学提供
        │
        ▼
[Vue 前端] ← B 同学提供登录页 + Axios + 路由守卫
```

## 四、API 接口契约

> **这是前期最重要的协作约定。** B、C、D 三人各自开发，接口必须提前约定清楚，避免联调时才发现不一致。以下接口为前期必须实现的最小集合。

### 4.1 统一约定

- 基础路径：`/api`
- 请求 Content-Type：`application/json`
- 需认证的接口在 Header 中携带 `Authorization: Bearer <jwt_token>`
- 统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { },
  "timestamp": 1710000000000
}
```

- 分页响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "content": [ ],
    "totalElements": 500,
    "totalPages": 25,
    "number": 0,
    "size": 20
  }
}
```

- 错误码约定：

| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 参数错误 |
| 401 | 未认证 / Token 过期 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

### 4.2 同学 B 负责接口：用户认证

### 7.1 项目结构

### 7.2 import_data.py 规格

```
输入: 命令行参数指定 CSV 文件路径，或从 config.py 读取默认路径
处理流程:
  1. pandas.read_csv() 读取文件
  2. 遍历每一行，按列映射规则转换为标准 JSON（格式见 7.4）
  3. json.dumps() 序列化
  4. redis_client.lpush('queue:raw-job-data', json_str)
  5. 每 100 条打印一次进度
  6. 完成后打印总计导入条数

列映射规则（CSV 列名 → JSON 字段）:
  不同数据集的列名不同，映射配置应放在 config.py 的 COLUMN_MAPPING 字典中
  示例:
    'Job Title'       → title
    'Company Name'    → company.name
    'Salary'          → 需要解析成 salary.min / salary.max
    'Location'        → city
    'Qualification'   → education
    'Experience'      → experience
    'Skills'          → skills (可能是逗号分隔的字符串)
    'Job Description' → 从中提取技能关键词
```

### 7.3 etl_clean.py 规格

```
运行模式: 常驻进程，while True 循环 BRPOP 阻塞等待
消费队列: queue:raw-job-data（BRPOP 超时时间 5s）
生产队列: queue:cleaned-job-data（LPUSH 每条清洗后的 JSON）

单条处理流程:
  1. json.loads() 解析
  2. 空值校验: 如果 title 或 company.name 为空 → 丢弃，count 记录
  3. 薪资标准化:
     - 如果已经是 {min, max} 格式且单位为 K → 直接使用
     - 如果是 "8K-15K" 字符串 → 正则提取
     - 如果是 "面议" 或无法解析 → salary_min=0, salary_max=0
     - 如果是年薪（>100）→ 除以 12 转月薪
     - 统一单位: K
  4. 城市标准化:
     - 从 city_mapping.json 查找城市名 → 补充 province 和 city_tier
     - 找不到的标记为 province="未知", city_tier="其他"
  5. 学历标准化:
     - 映射为枚举: 不限/大专/本科/硕士/博士
     - 模糊匹配: "本科及以上"→"本科", "Bachelor"→"本科"
  6. 经验标准化:
     - 映射为枚举: 不限/应届/1-3年/3-5年/5-10年/10年以上
     - 模糊匹配: "三年以上"→"3-5年", "一到三年"→"1-3年"
  7. 技能提取:
     - 从 skill_dict.json 加载 500+ 技术关键词
     - 在 title + description 字段中做关键词匹配（忽略大小写）
     - 去重后放入 skills 数组
     - 如果原始数据已有 skills 字段，合并去重
  8. 去重:
     - 计算 source_md5 = MD5(jobId + sourceUrl)
     - 查询 MySQL job_position 表 WHERE source_md5 = ?
     - 如果已存在 → 跳过（count 记录为 duplicate）
  9. 写入:
     - INSERT INTO job_company（如果 company_name 不存在则新增，存在则复用 id）
     - INSERT INTO job_position（使用上一步的 company_id）
     - LPUSH queue:cleaned-job-data 清洗后的完整 JSON
  10. 每条处理完打印日志: [时间] 处理 jobId=xxx title=xxx status=OK/SKIP/DUP

统计输出（每 100 条或退出时打印）:
  总消费 / 成功写入 / 空值丢弃 / 重复跳过 / 异常失败
```

## 十一、前期常见问题与注意事项

| 问题 | 预防措施 |
|------|---------|
| CSV 列名千奇百怪，导入脚本写死列名 | `config.py` 中用字典做列映射，换数据集只需改配置 |
| ETL 技能提取不准确 | 先用 500 词典跑一遍样本数据，人工抽查 50 条看技能字段是否合理，补充缺失关键词 |
| 薪资解析失败率高 | 优先挑选薪资格式规整的数据集；对于"面议"类数据，薪资设为 0 而非丢弃 |
| 四人 JPA 实体冲突 | 前期在 common 模块统一实体基类 `BaseEntity`（id + createTime + updateTime），各模块实体放自己模块 |
| Maven 多模块依赖混乱 | 父 POM 用 `<dependencyManagement>` 统一版本，common 模块不要依赖任何业务模块 |
| Docker 构建慢 | Spring Boot 和 Vue 都用多阶段构建，利用 Docker 层缓存 |
| 前端路由守卫逻辑复杂 | 前期只判断"有没有 Token"，不做权限细分；中期在 Pinia 中扩展 permissions 判断 |
| 不同城市名称格式不统一 | `city_mapping.json` 同时收录"北京""北京市""Beijing"三种写法，映射到同一标准名 |
