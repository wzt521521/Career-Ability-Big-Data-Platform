# v1.0.0 版本规则

> 状态：已确定
>
> 生效日期：2026-07-14
>
> 关联任务：[`R0-06`](https://github.com/wzt521521/Career-Ability-Big-Data-Platform/issues/10)

## 1. 版本口径

项目采用 Semantic Versioning。Maven、npm、容器镜像、Git tag 和 GitHub Release 必须表达同一个发行版本，不允许前后端或发布物使用互相冲突的版本号。

| 阶段 | Maven / npm 版本 | 镜像标签 | Git tag | 放行条件 |
| --- | --- | --- | --- | --- |
| 集成开发 | 后端暂用 `1.0.0-SNAPSHOT`；当前前端版本不作为正式发行声明 | 不发布正式标签 | 无 | 仅进入发行集成分支 |
| 首个 RC | `1.0.0-rc.1` | `1.0.0-rc.1` | `v1.0.0-rc.1` | 所有 P0 完成 |
| 后续 RC | `1.0.0-rc.N` | `1.0.0-rc.N` | `v1.0.0-rc.N` | 阻断缺陷修复后递增 N 并全量复验 |
| 正式版 | `1.0.0` | `1.0.0` | `v1.0.0` | 所有 P0/P1 和正式发行门禁通过 |

## 2. 一致性要求

- RC 阶段同时修改 `backend/pom.xml` 和 `frontend/package.json`，不得只改一端。
- 后端、前端和 Python ETL 镜像使用相同版本标签；验收和回滚不得依赖可移动的 `latest`。
- Git tag 必须指向已通过 required checks 的受保护分支提交，并使用签名标签。
- GitHub Release、变更日志、SBOM、校验和和部署说明引用同一个 Git tag。
- 正式版本发布后，兼容修复递增补丁版本；向后兼容功能递增次版本；破坏性变更递增主版本。

## 3. 发行顺序

1. 集成阶段保留后端 `1.0.0-SNAPSHOT`，禁止创建正式 tag 或正式镜像标签。
2. 所有 P0 关闭后执行 REL-01，将后端、前端和镜像统一改为 `1.0.0-rc.1`。
3. RC 发现阻断缺陷时修复并递增为 `rc.2`、`rc.3`，每次重新执行完整验收。
4. 所有 P0/P1 关闭后执行 REL-03，将版本统一改为 `1.0.0`，合入 `main` 并创建签名 tag `v1.0.0`。
5. 从正式 tag 重新构建并验证发布物，不能使用未提交文件或开发者机器残留产物。

## 4. 版本变更审计

- 每次版本变更必须关联 `REL-*` Issue 和独立 PR。
- PR 验证必须同时输出 Maven、npm、镜像和 Git tag 预期值。
- 版本号不一致视为发行阻断，不允许通过手工重命名产物规避。
