## 2. 审计结论

当前代码已经形成主要业务模块，但不满足可编译、可恢复、可持久化和可安全发行的最低条件。共记录 18 项问题：P0 6 项、P1 10 项、P2 2 项。

最关键的结论是：认证基础链路相对完整；数据、推荐、报告和开放 API 链路均存在至少一个阻断点。当前不能创建发行候选版本。

## 3. P0 阻断问题

### AUD-001 后端主线无法编译

**证据：** `Career-Ability-Big-Data-Platform/backend/src/main/java/com/career/platform/recommend/service/RecommendService.java:120` 和 `:163`。

`Set<String>` 调用了不存在的 `retain(...)`。`mvnw.cmd test` 在编译 153 个源文件时失败，后端测试、打包和 Docker 后端镜像都无法继续。

**处理：** 改为正确的集合交集操作并增加推荐算法测试。对应任务 R1-01。

### AUD-002 数据管道测试会无条件删除共享数据

**证据：** `data-pipeline/test_pipeline.py:13-16` 在模块加载时连接 Redis 并删除两个队列；`:36-47` 关闭外键检查并截断 `job_position`、`job_company`。

文件名会被 pytest 自动收集，但没有 fixture、测试数据库校验或显式确认。任何误执行都可能清空开发/验收数据。

**处理：** 立即拆分纯单元测试和隔离集成测试，禁止导入副作用和共享数据库 `TRUNCATE`。对应 R1-05、A-02。

### AUD-003 ETL 在异常时永久丢失原始消息

**证据：** `data-pipeline/etl_clean.py:407-415` 使用 `BRPOP` 先从 raw queue 删除消息；解析失败在 `:417-422` 直接丢弃；数据库或 Redis 异常只在 `:489-495` 回滚 MySQL，没有重新入队、处理中队列或死信队列。

该链路是 at-most-once，不满足“失败重试”和可恢复数据管道要求。进程在 MySQL commit 前崩溃会丢原始消息；在 commit 后、cleaned queue 写入前崩溃则数据库与 cleaned queue 不一致。

**处理：** 使用 processing queue + ack/requeue/DLQ，或 Redis Streams consumer group；数据库幂等键作为最终边界。对应 A-04、A-06、A-07。

### AUD-004 报告异步任务存在提交前读取竞态

**证据：** CodeGraph 路径为 `ReportServiceImpl.generate -> AsyncReportGenerator.generate -> AnalyticsService.calculateSnapshot`。`ReportServiceImpl.java:60-75` 在类级事务中保存记录后立即调用 `@Async`；异步线程在 `AsyncReportGenerator.java:67-73` 马上按 ID 查询。

外层事务可能尚未提交，异步线程会查不到记录并直接返回，数据库记录永久停在 `PENDING`，没有重试或恢复器。

**处理：** 事务提交后再发布任务，并增加 PENDING/GENERATING 超时恢复。对应 D-04。

### AUD-005 Docker Redis 缓存模式无法序列化推荐结果

**证据：** `docker-compose.yml:71` 强制 `SPRING_CACHE_TYPE=redis`；`application.yml:27-31` 未配置自定义序列化；`RecommendationResponse.java:10` 未实现 `Serializable`；`RecommendService.java:80-84` 将完整列表写入 Cache。

Spring Boot 2.7 默认 Redis cache value 使用 JDK 序列化。首次计算推荐后执行 `cache.put` 会因 DTO 不可序列化失败，本地 simple cache 测试无法发现该问题。

**处理：** 配置受控 JSON 序列化或可版本化缓存 DTO，并在真实 Redis 集成测试验证 TTL、读写和兼容性。对应 D-10。

### AUD-006 报告文件不持久化，容器重建即丢失

**证据：** `AsyncReportGenerator.java:43` 默认写 `./upload/reports`；`docker-compose.yml` 仅定义 MySQL、Redis volume，后端没有报告 volume。

数据库会保留 `COMPLETED` 和文件路径，但容器重建后文件消失，历史报告全部变成不可下载状态。

**处理：** 配置受控绝对目录和持久卷，验证重建、备份、恢复和删除。对应 D-07、I-03。

## 4. P1 高优先级问题

### AUD-007 报告标题可注入 HTML，可能触发服务端资源访问

**证据：** `GenerateReportRequest.java:16-24` 仅校验标题非空，没有长度或字符约束；三份 `.ftl` 在 `<title>`、`<h1>` 中直接使用 `${reportTitle!""}`，模板没有 HTML auto-escape 声明；`AsyncReportGenerator.java:102-106` 使用默认 `ConverterProperties` 转 PDF。

拥有 `report:view` 的用户可把 HTML 标签写入报告标题。绝对 `http/file` 资源标签可能被 PDF 转换器读取，形成 HTML 注入、SSRF 或本地资源暴露风险。

**处理：** 开启 Freemarker HTML 自动转义、限制标题长度/字符，并限制 pdfHTML resource retriever。增加恶意标题测试。

### AUD-008 报告时间范围只是展示文本，统计仍为全量数据

**证据：** `AsyncReportGenerator.java:81` 无条件调用 `calculateSnapshot()`；`:85-88` 仅把日期字符串放入模板。CodeGraph trace 显示 `calculateSnapshot -> positions -> findAllWithCompany`。

用户生成“7 月报告”时得到的是全库统计，却在 PDF 上标注 7 月范围，属于会误导用户的业务正确性问题。

**处理：** 复用 `AnalyticsFilter` 或新增报告快照契约，按起止日期和模板维度查询。对应 C-03、D-05。

### AUD-009 推荐没有执行全局 TOP20，分页参数也未校验

**证据：** `RecommendService.java:90-136` 对全部岗位排序后原样返回；`:142-145` 总数返回全部岗位数；`RecommendController.java:35-42` 接受任意 page/size。

这与需求中的 TOP20 不一致。`page <= 0`、`size <= 0` 会触发负下标、非法分页或异常响应。

**处理：** 在计算层截断 TOP20，统一总数，增加 Bean Validation 和边界测试。对应 D-01。

### AUD-010 功能权限和数据范围没有真正落到业务查询

**证据：** `DataScopeProvider` 只有接口、实现和测试，没有业务调用者；岗位、统计、分析、大屏 Controller 使用 `isAuthenticated()`，没有使用已定义的 `position:view`/`dashboard:view`；报告生成和删除也只使用 `report:view`。

前端隐藏菜单不能构成后端权限边界。五角色的 `ALL/COLLEGE/SELF` 当前只是未消费的契约。

**处理：** 明确业务数据范围并在 Service/Repository 过滤，拆分读/生成/删除权限。对应 B-01、B-02。

### AUD-011 采集管理与 Python 数据管道完全断开

**证据：** `CollectTaskService.java:31-54` 只保存 status、cron 和 retry 字段；仓库唯一业务 `@Scheduled` 位于 `AnalyticsScheduler.java:33`；没有任务执行器、进程调用、队列发布、启停或退避逻辑。

管理页面可以修改“任务状态”，但不会产生任何数据，属于表面闭环。

**处理：** 实现调度执行器、状态机、手动触发、取消、日志和重试。对应 A-03、A-04。

### AUD-012 开放 API 安全链闭合，但没有业务数据接口

**证据：** CodeGraph 确认 `ApiKeyAuthenticationFilter -> rate limiter -> JWT filter -> call log` 链路存在；实际 `/api/open/v1/**` Controller 只有 `OpenApiStatusController.java:15-22` 的 `/platform/status`。

岗位、技能、城市、报告、推荐等需求接口没有接入，因此第三方只能验证认证，不能消费平台数据。

**处理：** 补 B/C/D 业务开放接口并复用现有双认证、限流和审计。对应 B-03、B-04。

### AUD-013 离线统计表是写入终点，没有被查询层消费

**证据：** `OfflineAnalysisService.java:50-68` 写七张 `stat_*` 表；`AnalyticsService.java:280-282` 的查询入口仍调用 `findAllWithCompany()`，没有 stat repository/query。

定时任务没有降低页面查询成本；每个未命中缓存的统计请求和报告都把全部岗位及企业加载进 JVM。数据增长后会出现内存和延迟问题。

**处理：** 明确实时/离线读模型，页面优先读预计算结果，筛选查询下推数据库。对应 C-02、C-05。

### AUD-014 Redis 和 MySQL 的默认部署边界不适合发行

**证据：** `redis/redis.conf:4-6` 绑定 `0.0.0.0` 且关闭 protected mode；Compose 映射 Redis 6379；`docker-compose.yml:67-68` 后端使用 root 和开发默认密码；`application.yml:13-14,57` 还提供 root/root 和开发 JWT 默认值。

**处理：** 生产 override 不暴露 Redis/MySQL，Redis 启用认证或仅内部网络，应用使用最小权限数据库账号，生产缺密钥时启动失败。对应 I-04、B-06。

### AUD-015 Compose 不能满足“五服务 healthy”和数据/报告闭环

**证据：** `docker-compose.yml` 只有 MySQL、Redis 两个 healthcheck；ETL 没有挂载 `./data`，后端没有报告 volume；当前机器也未安装 Docker，尚无实跑证据。

**处理：** 为后端、前端、ETL增加健康/就绪判定，挂载样本和报告卷，在干净 Docker 环境执行全链路。对应 A-01、A-08、I-03、I-06。

### AUD-016 仓库跟踪 Python 字节码，测试依赖不可复现

**证据：** Git 跟踪 3 个 `data-pipeline/__pycache__/*.pyc`；`requirements.txt` 只使用无上界 `>=`，没有 pytest。

字节码与 Python 版本和执行环境绑定，本次基线同步时已有一个 `.pyc` 因 Python 3.13 运行被重写。它不应作为源代码或发行输入。

**处理：** 删除已跟踪字节码，补 ignore、测试依赖和锁定/约束文件。对应 R1-06。

## 5. P2 改进项

### AUD-017 Redis 限流的递增和 TTL 设置不是一个原子操作

`RedisApiRateLimiter.java:19-30` 先 increment，再在 count=1 时 expire。两步之间进程崩溃会留下无 TTL 的历史分钟 Key。Key 含分钟，不会持续阻断当前请求，但会形成无界垃圾 Key。

**处理：** 用 Lua 将 INCR/EXPIRE 原子化。

### AUD-018 依赖维护和许可证需要发行决策

后端使用 Spring Boot 2.7.18；PDF 使用 `com.itextpdf:html2pdf:5.0.5`。需要完成停止维护风险登记、漏洞扫描、iText AGPL/商业许可证确认，并在 Alpine 镜像配置可再分发中文字体。

**处理：** 对应 D-08、D-09、S-02、S-06。
